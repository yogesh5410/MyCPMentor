§1 — Overview

Project Overview & Goals

MyCPMentor is an AI-Powered Competitive Programming Operating System — a unified platform that learns from a programmer's history, prescribes targeted practice, runs competitions, and adapts recommendations in real time.

What Is MyCPMentor?

MyCPMentor is a full-stack web platform built for competitive programmers who want structured, intelligent improvement rather than random grinding. The platform combines:

  • Curated Problem Pool — sourced from Codeforces (scraped) and AI-generated custom problems tailored to a user's weak spots.
  • AI Mentor — an LLM-powered agent that reads a user's full solve history, CF data, and battle results to give hyper-personalized guidance.
  • Live Testing — an in-browser code editor backed by a sandboxed test-runner service with real test cases and custom test support.
  • Competitive Layer — 1v1 battles, platform-hosted contests, and a composite rating system.
  • Browser Extension — solves problems directly on codeforces.com while silently syncing submissions back to MyCPMentor for analytics.

Core Design Principles

🎯 Context-Aware
Every recommendation, POTD selection, and AI Mentor message is personalized to the individual user's current level, recent mistakes, and stated goals — not generic.
🔗 CF-Native
Codeforces is the world's largest CP platform. MyCPMentor syncs a user's CF history and scrapes CF's full problem bank — users don't leave their existing workflow.
⚡ Real Feedback Loop
Submissions run against real test cases. Wrong answers show which test case failed. TLE analysis shows which constraint is the bottleneck.
📊 Data-Driven Progress
Every solve, battle, contest, and revision session feeds into a composite rating and analytics pipeline. Progress is measured and visualized — not assumed.

User Journey

1
Register & Connect CF
User signs up, links their Codeforces handle. Backend pulls full submission history via CF API and populates their solve map.
2
AI Onboarding Assessment
AI Mentor analyzes CF history: strongest/weakest tags, rating trajectory, plateau patterns. Generates a personalized roadmap.
3
Daily Practice Loop
POTD surfaces a relevant problem. User solves it in the in-browser editor or via the CF extension. Submission is judged in the test-runner.
4
Battle & Contest
User challenges friends or random opponents to live coding battles. Participates in platform contests. Rating updates in real time.
5
Review & Iterate
Analytics surfaces error patterns. AI Mentor generates weekly review. Revision queue resurfaces unsolved or incorrectly-solved problems via spaced repetition.

§5 — Problem Ecosystem

CF Problems — Sourcing, Scraping & Storage

Codeforces has 10,000+ problems. MyCPMentor mirrors this pool using the official CF API for metadata and a custom scraper for problem statements and test cases.

Data Sources

📡 Codeforces API v1
Official JSON API. Endpoints used:
  • problemset.problems — all problems with tags, ratings, titles
  • contest.standings — contest result data
  • user.status — a user's submission history
  • user.rating — CF rating history
Rate limit: 1 req/2s per IP. Use server-side caching aggressively.
🕷️ Custom HTML Scraper
The CF API does NOT provide problem statements, editorial text, or sample test cases in structured form. A Playwright/Puppeteer scraper:
  • Navigates to each problem URL
  • Extracts HTML statement (converted to Markdown)
  • Scrapes sample input/output pairs
  • Downloads time/memory limits

CF Scraper Pipeline

1
Bootstrap Sync (one-time)
On initial setup, call problemset.problems to fetch all ~10,000 problem stubs. Store in MongoDB problems collection with source: "cf", cfContest, cfIndex, tags[], rating. Mark as scraped: false.
2
Statement Scrape (batch worker)
A Bull queue job processes unscraped problems in batches of 50. Playwright headless browser fetches codeforces.com/problemset/problem/{contest}/{index}. Extracts .problem-statement div → convert to Markdown. Stores in problem.statement. Marks scraped: true.
3
Test Case Acquisition
Sample test cases are scraped from problem page HTML. Full test cases are obtained via: (a) CF Polygon API if the problem is open-source, (b) community-submitted tests from existing accepted solutions, (c) AI-generated edge cases validated against a known correct solution.
4
Incremental Sync (daily cron)
A cron job runs every 24h, calls CF API for problems added since last sync timestamp. New problem stubs are inserted and queued for scraping. The scraper worker picks them up automatically.
5
Redis Cache Population
After scraping, hot problems (rating 800–2400, solve count >1000) are pre-loaded into Redis with a 24h TTL. POTD candidates are cached separately keyed by date.

§6 — Problem Ecosystem

AI-Generated Problems — Generation Pipeline

When a user needs practice on a specific tag at a specific difficulty and CF doesn't have enough unsolved examples, the AI generates a custom problem with full statement, test cases, and editorial.

Trigger Conditions

The AI Problem Generator fires in two scenarios:

  • On Demand — User clicks "Generate Problem" in Practice or Sheets with a target tag + difficulty.
  • AI Mentor Prescription — AI Mentor detects a gap (e.g. user struggles with DP on Trees) and auto-queues an AI problem of appropriate difficulty + style.

Generation Pipeline

1
Request to AI Service
API Gateway routes POST /ai/generate-problem with payload: { userId, tags: ["dp", "trees"], rating: 1800, avoidProblemIds: [...] }
2
Prompt Engineering
AI Service builds a structured prompt: "Generate a competitive programming problem at ~1800 Codeforces difficulty. Topic: DP on Trees. The problem must have a non-trivial observation. Provide: title, story, input/output spec, constraints, 3 sample test cases, 5 hidden test cases with answers, time complexity of solution, full editorial with C++ code."
3
LLM Response Parsing
Response is streamed and parsed into structured JSON using Zod schema validation. Fields: title, statement, inputSpec, outputSpec, sampleTests[], hiddenTests[], editorial, solutionCode.
4
Self-Verification
The AI-generated solution code is run through the Test Runner against all generated test cases. If the solution fails any case, the problem is re-sent to the LLM for correction (max 2 retries). If it still fails, the problem is flagged for admin review and a CF problem is served as fallback.
5
Storage & Serving
Verified problems are stored in MongoDB with source: "ai", generatedFor: userId, isPublic: false (unless admin promotes it). They never appear in other users' practice queues unless explicitly shared.

AI Problem Document Schema

{
  _id: ObjectId,
  source: "ai",
  generatedFor: UserId,          // personalized to this user
  tags: ["dp", "trees"],
  rating: 1800,
  title: "The Tree Merchant",
  statement: "...",
  inputSpec: "...",
  outputSpec: "...",
  constraints: "1 ≤ n ≤ 2×10^5",
  sampleTests: [...],
  hiddenTests: [...],
  editorial: "...",
  solutionCode: "...",           // AI's reference solution (C++)
  isVerified: true,              // passed self-test
  isPublic: false,
  adminStatus: "pending" | "approved" | "rejected",
  generationModel: "gpt-4o",
  createdAt: Date
}
⚠️ Test Case Integrity AI-generated test cases must pass through the AI's own solution before storage. The test runner runs the solutionCode against all hiddenTests and if any mismatch is found, the problem is invalidated. This prevents users from encountering unsolvable or incorrectly specified problems.
§7 — Extension

Browser Extension Architecture

The MyCPMentor extension runs inside codeforces.com — injecting a problem sidebar, syncing submissions to MyCPMentor, and letting users solve CF problems while getting real-time AI hints without leaving the CF interface.

Extension Components

📄 manifest.json
Manifest V3. Declares host_permissions for codeforces.com/* and the MyCPMentor API URL. Registers content.js for CF pages and background.js as service worker.
🔧 content.js (Content Script)
Injected into every codeforces.com/problemset/problem/* and codeforces.com/contest/*/problem/* page. Reads DOM to extract: problem title, contest ID, index. Injects the MyCPMentor sidebar UI.
⚙️ background.js (Service Worker)
Handles API calls to MyCPMentor backend (avoids CORS issues from content scripts). Listens for messages from content script and relays to API. Manages JWT token storage in chrome.storage.local.
🖼️ Popup UI (React)
Extension popup (click icon): shows login status, current streak, today's POTD, and quick link to dashboard. Built with React + Vite, bundled separately.

Auth Flow — Shared JWT with Website

1
Login On Website
User logs in on MyCPMentor web app. JWT stored in localStorage.
2
Extension Reads Token
On extension install/popup open, content script sends chrome.runtime.sendMessage({ type: 'GET_TOKEN' }). Background service worker checks chrome.storage.local. If not found, redirects user to login in popup.
3
Token Sync
The MyCPMentor web app detects if the extension is installed (via chrome.runtime.sendMessage handshake). If yes, it posts the JWT to the extension via window.postMessage → content script → background → chrome.storage.local.
4
API Calls from Extension
All API calls from the extension include the JWT as Authorization: Bearer <token>. The backend treats extension calls identically to web app calls — the same JWT verify middleware runs.

Submission Interception on CF

When a user submits code on codeforces.com, the content script intercepts the submission:

  1. Content script adds a MutationObserver on the CF verdict element (#pageContent .verdict).
  2. When a verdict appears (AC/WA/TLE), the content script reads the problem ID and verdict.
  3. It sends { problemCfId, verdict, language, submissionId } to the background worker.
  4. Background worker calls POST /cf/submission-sync on MyCPMentor backend with the JWT.
  5. Backend saves the submission to MongoDB, updates the user's solve map, and triggers rating recalculation if applicable.
💡 In-Page Sidebar Features The injected sidebar shows: ① Difficulty and tags from MyCPMentor DB, ② "Add to Sheet" button, ③ AI Hint button (costs 1 hint token), ④ User's previous attempts on this problem, ⑤ "I solved it" manual override for offline solutions.
§8 — Test Runner

Test Case Runner Service

The Test Runner is an isolated microservice that executes user-submitted code in sandboxed Docker containers, compares output against expected answers, and returns structured verdicts.

Architecture Overview

API Gateway │ │ POST /submissions { code, language, problemId, userId } ▼ ┌─────────────────────────────────┐ │ Submission Controller │ │ 1. Validate + sanitize input │ │ 2. Load test cases from DB │ │ (or Redis cache) │ │ 3. Enqueue to Bull queue │ │ "submission-queue" │ │ 4. Return { jobId } │ └──────────────┬──────────────────┘ │ Bull Job ▼ ┌─────────────────────────────────┐ │ Test Runner Worker │ │ (Bull consumer, N concurrent) │ │ │ │ For each test case: │ │ ┌─────────────────────────┐ │ │ │ docker run │ │ │ │ --memory=256m │ │ │ │ --cpus=1 │ │ │ │ --network none │ │ │ │ --read-only │ │ │ │ judge-runner:cpp17 │ │ │ │ [compiled binary] │ │ │ └─────────────────────────┘ │ │ Compare stdout vs expected │ │ Emit verdict via WebSocket │ └─────────────────────────────────┘

Verdict Types

VerdictCodeCondition
AcceptedACstdout matches expected output (ignoring trailing whitespace)
Wrong AnswerWAOutput differs from expected on test N
Time Limit ExceededTLEProcess doesn't finish within time limit × 1.5 (kill margin)
Memory Limit ExceededMLEContainer RAM usage exceeds limit
Runtime ErrorRENon-zero exit code (segfault, exception, etc.)
Compilation ErrorCECompiler returns errors — returned immediately, no execution
Presentation ErrorPEOutput whitespace/formatting differs but values are correct

Security Sandbox

  • Network isolation: --network none — container cannot make any network calls.
  • Read-only filesystem: --read-only — cannot write to filesystem except /tmp.
  • User namespace: Container runs as non-root user (UID 65534).
  • Seccomp profile: Syscall allowlist restricts to read/write/exec only. No fork bombs.
  • Resource limits: CPU = 1 core, RAM = 256MB, disk write = 0 (read-only), time = problem TL × 1.5.
  • Container reuse: Pre-warmed container pool of 10 per language reduces cold-start latency from ~800ms to ~50ms.

Streaming Verdicts to Frontend

Users see verdicts as they arrive (test by test), not as a batch at the end. This uses Server-Sent Events (SSE):

  GET /submissions/:jobId/stream
  Content-Type: text/event-stream

  data: {"testCase": 1, "verdict": "AC", "time": 23, "memory": 12}
  data: {"testCase": 2, "verdict": "AC", "time": 31, "memory": 14}
  data: {"testCase": 3, "verdict": "WA", "expected": "6", "got": "5"}
  data: {"done": true, "finalVerdict": "WA", "passedTests": 2, "totalTests": 10}

Switching Flow in Practice Mode

1
User Opens Practice
Default shows CF problems filtered by: user's current weak tags + rating ± 200 from their current MyCPMentor rating. Source toggle shows "CF" (active) and "AI" buttons.
2
Source Toggle → "AI"
Frontend calls GET /problems?source=ai&tags=dp,trees&rating=1800. Backend checks if AI problems exist for these params for this user. If not, triggers generation pipeline in background. Returns a loading state with ETA.

The problem page renders identically regardless of source. Problem metadata includes source field — a small "AI" or "CF" badge is shown in the top-right. The in-browser editor, test runner, and verdict display work identically.

Submissions against AI problems are stored in submissions collection with problemSource: "ai". They count towards rating, analytics, and revision scheduling — no difference from CF problems in the data pipeline.

For CF problems, users have a choice of two solving environments:

🖥️ In-Browser Editor (Default)
Monaco editor embedded in MyCPMentor. Code submitted to our Test Runner. Verdict shown inline. Full analytics captured. No CF account needed.
🌐 Solve on CF (Extension)
Button opens the problem on codeforces.com. Extension detects the problem and injects the MyCPMentor sidebar. When user submits on CF, extension intercepts the verdict and syncs it back to MyCPMentor.

Cache-Aside Pattern (Standard Read Flow)

async function getProblem(problemId) {
  const cacheKey = `problem:${problemId}`;

  // 1. Check Redis
  const cached = await redis.get(cacheKey);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss — read MongoDB
  const problem = await Problem.findById(problemId).lean();
  if (!problem) throw new NotFoundError();

  // 3. Populate cache
  await redis.setex(cacheKey, 3600, JSON.stringify(problem));

  return problem;
}

Cache Invalidation Strategy

  • Problems — Invalidated on admin edit: redis.del(`problem:${id}`)
  • User Session — Invalidated on profile update or submission record
  • Leaderboard — Rely on TTL (5min); acceptable staleness
  • POTD — Replaced atomically by cron: SET potd:2026-04-14:easy ... EX 86400
  • Contest Standings — Short TTL (30s) with write-through on AC submission
💡 Pre-warming Hot Problems After any CF scraper sync, a post-sync worker pre-warms the Redis cache for the 1000 most-solved CF problems (rating 800–2000, solve count > 5000). This ensures the most frequently accessed problems are always served from cache.

POTD (Problem of the Day) System

POTD surfaces one curated problem per difficulty (Easy/Medium/Hard) per day.

POTD Selection Algorithm

1
Nightly Cron (00:00 UTC)
A cron job runs each night to select tomorrow's POTD. It calls an AI Service endpoint that picks the best problem for each difficulty tier.
2
AI Selection Criteria
For each difficulty tier, AI ranks candidate problems by: (a) not solved by majority (<30% user solve rate), (b) covers tags that platform users are collectively weak at (aggregate analytics), (c) hasn't appeared as POTD in last 90 days, (d) has complete statement and test cases.
3
Admin Override
Admin can preview the AI selection 7 days ahead and override any day via the Admin → Problem Bank → POTD Scheduler. Overrides take priority.
4
Cache Population
Selected POTDs are written to Redis: SET potd:2026-04-15:easy {problemJson} EX 86400. All user requests are served from Redis — MongoDB is not hit for POTD reads during the day.
📊 POTD Personalization (v2) Future version: Each user sees a POTD personalized to their level. Redis key changes to potd:{date}:{userId}. Generated per-user by AI Mentor at midnight based on individual analytics. Requires pre-computation for all active users — runs as a Bull job fan-out.

§23 — Features

Battles — 1v1 Real-Time Coding

Battles are live, real-time 1v1 coding competitions. Both players get the same problem and the first to submit AC wins. Uses WebSocket for real-time synchronization.

Battle Lifecycle

1
Challenge Creation
Player A clicks "Challenge Friend" or "Find Opponent" (rated matchmaking). POST /battles/challenge { opponentId?, rating, tags? }. Server creates a Battle doc with status: "waiting" and emits a Socket.io event to opponent.
2
Opponent Accepts
POST /battles/:id/accept. Server assigns a problem (filtered by both players' rating average ± 100). Sets status: "active", startedAt: now. Both players receive the problem via Socket.io event battle:start.
3
Live Battle
Both players code in their editors. Submissions go to Test Runner. Results are sent to both players via Socket.io in real-time: battle:opponent-submitted events show opponent activity (not their code — only that they submitted and the verdict).
4
First AC Wins
When first AC is received, server sets winnerId, status: "finished". Enqueues a rating-queue job to compute and apply rating deltas. Emits battle:result to both players.
5
Rating Update
Rating Worker calculates delta using Elo-variant: expected win probability based on rating difference → actual outcome → delta. Winner gains, loser loses. Max ±50 per battle. Updates users.mcmRating and appends to ratings collection.

Matchmaking Algorithm

// Find an opponent within ±200 rating who is "searching" status
async function findOpponent(userId, userRating) {
  const searchingUsers = await redis.smembers('battle:searching');
  for (const candidateId of searchingUsers) {
    const candidateRating = await redis.hget(`user:${candidateId}`, 'rating');
    if (Math.abs(candidateRating - userRating) <= 200 && candidateId !== userId) {
      await redis.srem('battle:searching', candidateId);
      return candidateId;
    }
  }
  // No match — add self to searching pool (expire after 60s)
  await redis.sadd('battle:searching', userId);
  await redis.expire(`battle:searching:${userId}`, 60);
  return null;
}

Analytics Pipeline

Analytics aggregates raw submission data into human-readable insights. Computed via MongoDB aggregation pipelines, cached in Redis for 10 minutes per user.

Key Aggregations

// Tag accuracy: for each tag, calculate AC rate
db.submissions.aggregate([
  { $match: { userId: ObjectId(userId) } },
  { $lookup: { from: 'problems', localField: 'problemId', foreignField: '_id', as: 'problem' } },
  { $unwind: '$problem' },
  { $unwind: '$problem.tags' },
  { $group: {
      _id: '$problem.tags',
      total: { $sum: 1 },
      accepted: { $sum: { $cond: [{ $eq: ['$verdict', 'AC'] }, 1, 0] } }
  }},
  { $addFields: { accuracy: { $divide: ['$accepted', '$total'] } } },
  { $sort: { accuracy: 1 } }  // weakest tags first
])

MyCPMentor Composite Rating Formula

The composite rating is calculated from 5 weighted components, recomputed after each relevant event:

MCM_Rating = Σ (component_score × weight) Components: ┌────────────────────┬────────┬─────────────────────────────────────────┐ │ Component │ Weight │ Calculation │ ├────────────────────┼────────┼─────────────────────────────────────────┤ │ Solve Speed │ 0.25 │ Avg time-to-AC vs expected for rating │ │ Difficulty Range │ 0.20 │ Weighted avg difficulty of AC problems │ │ Battle Performance │ 0.25 │ Elo rating from battles │ │ Contest Performance│ 0.20 │ Elo delta from contests │ │ Consistency │ 0.10 │ Solve streak * recency weight │ └────────────────────┴────────┴─────────────────────────────────────────┘ Rating Tiers: 0–999: Beginner | 1000–1399: Learner | 1400–1799: Competitor 1800–2199: Expert | 2200–2599: Master | 2600+: Grandmaster

Rating Worker

// Triggered after: battle end, contest end, daily solve
async function recalculateRating(userId) {
  const [solveComponent, battleComponent, contestComponent, consistencyComponent] =
    await Promise.all([
      calcSolveComponent(userId),
      calcBattleElo(userId),
      calcContestPerformance(userId),
      calcConsistency(userId)
    ]);

  const newRating = Math.round(
    solveComponent    * 0.25 +
    difficultyRange   * 0.20 +
    battleComponent   * 0.25 +
    contestComponent  * 0.20 +
    consistencyScore  * 0.10
  );

  await User.findByIdAndUpdate(userId, { mcmRating: newRating });
  await redis.publish(`rating:updated:${userId}`, JSON.stringify({ newRating }));

  // Save to rating history
  await Rating.create({ userId, rating: newRating, trigger: 'battle', timestamp: new Date() });
}

§30 — Auth

Auth & Session Flow

Auth Methods

📧 Email + OTP
1. User enters email. 2. Backend generates 6-digit OTP, stores in Redis with 10min TTL (otp:{email}). 3. OTP sent via email (Resend/SendGrid). 4. User enters OTP → verified → JWT issued. No stored passwords. OTP is rate-limited (3 attempts max per 10min).
🔵 Google OAuth 2.0
1. User clicks "Sign in with Google". 2. Redirect to /auth/google (Passport.js). 3. Google callback at /auth/callback. 4. Passport creates/updates User doc with googleId. 5. JWT issued. 6. Frontend redirect to /auth/callback?token=... → stored in localStorage.

Middleware Chain

// Applied to all /dashboard/* and /api/protected/* routes
const authMiddleware = async (req, res, next) => {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = await User.findById(decoded.userId).select('-passwordHash').lean();
    if (!req.user || req.user.isBanned) return res.status(403).json({ error: 'Forbidden' });
    next();
  } catch (err) {
    return res.status(401).json({ error: 'Token expired or invalid' });
  }
};

// Role guard for admin routes
const requireRole = (role) => (req, res, next) => {
  if (req.user.role !== role && req.user.role !== 'admin')
    return res.status(403).json({ error: 'Insufficient permissions' });
  next();
};

OTP Security Details

  • OTP is a cryptographically random 6-digit number: crypto.randomInt(100000, 999999)
  • Redis key: otp:{email}, TTL: 600 seconds (10 min)
  • Attempt counter: otp:attempts:{email} — auto-expires with OTP key
  • After 3 failed attempts: key deleted, user must request new OTP
  • OTP is never logged or returned in API responses
